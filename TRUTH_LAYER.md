## Core System Invariants

- UI data access uses `useAppQuery` / `useAppMutation` with `query-tiers`.
- Every new read path must follow `docs/system_map/CACHE_ARCHITECTURE.md` (tier class, scope-safe keying, invalidation-first freshness).
- Server-side auth context comes from `requireAuth()` and validated role checks.
- Org/workspace sensitive writes are protected by RLS or service-role actions with explicit authorization checks.
- Trigger.dev handles background and asynchronous workflows; debugging must include related tasks.
- Webhook handlers must preserve idempotency and bounded retries.
- First-party Shopify webhook ingress (`/api/webhooks/shopify`) is observe-only for inventory/order topics (`inventory_levels/update`, `orders/create`, `orders/updated`). ShipStation is authoritative for order/inventory movement; Clandestine Shopify product sync remains Graph API task-driven.
- Release confidence is enforced by `release-gate` checks and full-site audit criteria.
- Bundle inventory uses component-level tracking via `bundle_components` table. When a bundle sells (any channel), `bundle-component-fanout` decrements each component SKU. Inventory pushes compute MIN(bundle_stock, floor(component_stock/qty)) for effective availability. `bundles_enabled` flag is advisory; `bundle_components` row existence is the source of truth.
- Bandcamp follows an **authority lifecycle**: Bandcamp API is authoritative for **initial ingest** (new titles, SKU/quantity/date/price bootstrap). After staff review or physical count, the warehouse app becomes authoritative for **operational fields** (SKU, quantity, price, dates). Bandcamp remains authoritative for **descriptive/external fields** (URL, subdomain, album_title, options, sales data) permanently. Governed by `authority_status` on `bandcamp_product_mappings` (`bandcamp_initial` → `warehouse_reviewed` → `warehouse_locked`). HTML scraping (`data-tralbum`) is **enrichment only** (about, credits, tracks, package photos). Automation is **bounded** (caps, DLQ to `warehouse_review_queue`, no unbounded retry).

- **ShipStation v2 inventory transaction-type semantics (Phase 0 — Patch D2 probe, 2026-04-17):** ShipStation v2 `POST /v2/inventory` enforces a `quantity >= 1` validator on every `transaction_type` that takes a `quantity`/`new_available` field — but the validator is applied asymmetrically across operations and lifecycle stages. Empirically verified contract (full evidence: plan §5.1, §7.1.6.0, `reports/shipstation-v2-decrement-to-zero-2026-04-17.json`):
  - `increment quantity: 0` on a fresh SKU → **400** (seed cannot create a zero-stock row).
  - `increment quantity: 1+` on a fresh SKU → **200** (canonical seed path).
  - `decrement quantity: N` on tracked SKU, including landing on `available: 0` → **200**. Use this for the natural delta path; **fanout MUST use `decrement` at the 1 → 0 boundary**, no special branch.
  - `adjust quantity: 0` on tracked SKU → **200** (asymmetric vs seed; this is the documented safety net if `decrement` ever fails).
  - `modify new_available: 0` → **400** (`modify` cannot zero a SKU). **Forbidden** in any inventory write path; Phase 4 fanout unit tests must assert this combination is never emitted.
  - When tracked `available` reaches 0, ShipStation drops the row from `GET /v2/inventory?sku=…` (returns empty `inventory: []`). Reconcile (Phase 5) MUST treat "DB has `available: 0`" and "ShipStation returns no row" as equal, not as drift.

Stale or empty tables emit alert-level log lines that the SLO doc maps to PagerDuty. The agent task does NOT exercise full sandbox restore — that is operator-led and documented in `docs/operations/runbooks/backup-verify.md`. Together, the agent probe (aliveness) and the operator runbook (restorability) constitute the full Tier 1 #8 contract.
- **Service-role usage is documented and CI-enforced (Tier 1 hardening #2 — 2026-04-13):** every `createServiceRoleClient()` call site in `src/**` is enumerated in `docs/security/service-role-usage.md` with a justification (Trigger task / webhook / OAuth handler / Server Action / shared lib). `scripts/check-service-role-usage.sh` (CI) greps the source tree for `createServiceRoleClient` (excluding tests + the helper itself), counts unique files, and fails the build if the count drifts from the manifest entry count. Adding a new service-role usage requires a manifest entry in the same PR — preventing accidental RLS bypass from sneaking into the codebase undocumented.
- **Tenant isolation is integration-tested (Tier 1 hardening #3 — 2026-04-13):** `tests/integration/tenant-isolation.test.ts` (run via `pnpm test:integration` against `vitest.integration.config.ts`) seeds two synthetic workspaces with overlapping data shapes via the service-role client, then attempts cross-workspace reads from the anon client authenticated as workspace-A's user. Asserts RLS denies access to workspace-B's `warehouse_products`, `warehouse_inventory_levels`, and adjacent org-scoped tables. The test skips gracefully if `SUPABASE_INTEGRATION_*` env vars are unset (so unit-test suites stay decoupled), and is wired into the release-gate when integration secrets are available.
- **Sentry tracing wraps the inventory fanout (Tier 1 hardening #4 — 2026-04-13):** `fanoutInventoryChange` opens a `Sentry.startSpan({ name: 'inventory.fanout', op: 'fanout' })` with `workspace_id`, `sku`, `delta`, and `correlation_id` as attributes, and each Shopify push is wrapped in a child span. Errors inside the span auto-capture to Sentry (Trigger.dev tasks already initialise `@sentry/node` per Rule #49). The span graph is the single observability handle for "what happened during a fanout" — no need to stitch together log lines across integrations.
- **Phase 5 tiered reconcile + canonical sync-state view (2026-04-13):** Steady-state safety net for the bidirectional bridge. Three schedule wrappers — `shipstation-bandcamp-reconcile-hot` (cron `*/5 * * * *`), `shipstation-bandcamp-reconcile-warm` (`*/30 * * * *`), `shipstation-bandcamp-reconcile-cold` (`0 */6 * * *`), all pinned to `shipstationQueue` (concurrencyLimit 1, shared with seed + SHIP_NOTIFY) — call the same inner runner `runShipstationBandcampReconcile(tier, payload, ctx, deps)` in `src/trigger/tasks/shipstation-bandcamp-reconcile.ts`. Tier candidate selection: hot = SKUs with `available <= HOT_LOW_STOCK_THRESHOLD (10)` UNION SKUs with `warehouse_inventory_activity` rows where `delta < 0` in the last 24h; warm = SKUs with `delta < 0` in the last 30 days; cold = full per-workspace catalog. Every tier excludes bundle parents (handled by Phase 2.5(c) `bundle-derived-drift-sensor`) and short-circuits with `skipped_no_v2_defaults` when `workspaces.shipstation_v2_inventory_warehouse_id` / `_location_id` are NULL. v2 reads use the batch-only `listInventory({ skus, inventory_warehouse_id, inventory_location_id })` (Rule §7.1.6 — NEVER per-SKU `getInventoryBySku`); a SKU at `available: 0` is invisible to v2 (Phase 0 §4.2.3) so a missing v2 row is treated as `available: 0`, NOT as drift. Drift thresholds (plan §7.1.6.1): `|drift| <= SILENT_DRIFT_TOLERANCE (1)` → silent auto-fix, no review item; `2 <= |drift| <= HIGH_SEVERITY_DRIFT_THRESHOLD (5)` → auto-fix + low-severity review queue row; `|drift| > 5` → auto-fix + high-severity review queue row (surfaced on the admin Channels page). Auto-fix is ALWAYS through `recordInventoryChange({ source:'reconcile', correlationId:'reconcile:{tier}:{run_id}:{sku}' })` — the Phase 5 schema migration (`20260413000030`) admits `'reconcile'` into `warehouse_inventory_activity_source_check`. v2 is the source of truth at this layer; reconcile NEVER writes to v2. Review queue dedup uses `group_key = 'reconcile.qty_drift:{workspace_id}:{sku}'` (Rule #55 — re-detection on the existing OPEN row bumps `occurrence_count` and may escalate severity low → high if drift widens). Per-tier `channel_sync_log` rows (`channel='shipstation_v2'`, `sync_type='reconcile_{tier}'`) drive the admin monitoring page (`/admin/settings/shipstation-inventory`). Companion view `sku_sync_status` (Phase 5 migration §7.1.13) denormalizes per-variant Bandcamp mapping state, baseline-anomaly status, sku-conflict status, last successful v2/Bandcamp push timestamps, and last external error into a single read so admin pages don't reconstruct truth from 6+ joins per request. Live view today; materialization deferred until any admin surface exceeds 500ms p95 (Open Question #9). The reconcile sensor and the Phase 4 focused legs are complementary: focused legs catch real-time changes; reconcile catches what focused legs drop.
- **ShipStation Inventory Sync echo contract (2026-04-13 second-pass audit):** The operator activated **ShipStation Inventory Sync** for every connected Shopify / Squarespace / WooCommerce store AND ShipStation has native Bandcamp store integrations registered in `warehouse_shipstation_stores` (10+ Bandcamp stores across 23 SS stores total at audit time). SS Inventory Sync subscribes directly to each storefront's order/inventory webhooks and decrements v2 natively at import time — completely independent of our app's webhook processing. Therefore the canonical echo-skip set in `src/lib/server/inventory-fanout.ts` (`SHIPSTATION_V2_ECHO_SOURCES`) MUST include EVERY storefront source that SS Inventory Sync mirrors: `shipstation` (SHIP_NOTIFY processor), `reconcile` (Phase 5 sensor), `shopify`, `squarespace`, `woocommerce`, `bandcamp`. Warehouse-side write sources (`manual`, `manual_inventory_count`, `cycle_count`, `inbound`, `preorder`, `backfill`) are NOT in the echo set — those originate in our app and v2 has not yet seen them, so they MUST fanout. The companion implication for sale polls: `bandcamp-sale-poll` does NOT enqueue `shipstation-v2-decrement` (the explicit Phase 4 trigger was removed in the second-pass audit — SS already decremented v2 when it imported the order). The `shipstation-v2-decrement` task code is preserved as a manual rollback path; if SS Inventory Sync is ever disabled per-workspace, BOTH the explicit enqueue in `bandcamp-sale-poll` AND the `'bandcamp'` membership in `SHIPSTATION_V2_ECHO_SOURCES` must be reverted together (never one without the other — the comments in both files cross-reference each other). Phase 5 reconcile remains the safety net for any drift the echo skip introduces.
- **Phase 4 bidirectional bridge (2026-04-13, amended 2026-04-13 second-pass audit):** Per-SKU cross-system inventory mirroring runs through two ledger-gated, queue-pinned tasks instead of inline calls. (1) **`shipstation-v2-decrement`** (`shipstationQueue`, concurrencyLimit 1): formerly triggered by `bandcamp-sale-poll` after each successful `recordInventoryChange()` for a Bandcamp sale; the explicit enqueue was removed in the second-pass audit (see "ShipStation Inventory Sync echo contract" above). Task code is preserved for manual rollback / future per-workspace activation. Correlation-id shape `sale:{band_id}:{package_id}:{newSold}:{sku}`. Skip cascade: `fanout-guard.shouldFanout("shipstation", correlationId)` → `workspaces.shipstation_v2_inventory_warehouse_id`/`_location_id` IS NOT NULL → variant resolved → variant is NOT a `bundle_components.bundle_variant_id` (Phase 2.5(a) bundle exclusion) → ledger acquired (`UNIQUE(system='shipstation_v2', correlation_id, sku, action='decrement')`). On the success path, calls `adjustInventoryV2({ transaction_type:'decrement', quantity:|delta| })` — NEVER `modify` (Phase 0 Patch D2 — `modify new_available:0` returns 400). On failure, marks the ledger row `error` and re-throws so Trigger.dev retries the task (the same correlation_id makes the retry a no-op once the prior attempt eventually succeeds elsewhere). (2) **`bandcamp-push-on-sku`** (`bandcampQueue`, concurrencyLimit 1, OAuth-serialized — Rule #9): triggered by `process-shipstation-shipment` after each per-line successful `recordInventoryChange()`. Correlation-id shape `ship:{shipmentId}:{sku}`. Skip cascade: `fanout-guard.shouldFanout("bandcamp", correlationId)` → variant resolved → NOT distro (`warehouse_products.org_id IS NOT NULL`) → variant is NOT a bundle parent → `bandcamp_product_mappings` row exists → `push_mode IN ('normal','manual_override')` (Phase 1 contract) → mapping is package-level (option-level mappings defer to the 5-min `bandcamp-inventory-push` cron) → ledger acquired (`UNIQUE(system='bandcamp', correlation_id, sku, action='modify')`). On the success path, computes `pushed_quantity = MAX(0, available - effective_safety)` from `warehouse_inventory_levels` (identical formula to the cron path so cron + focused push agree), refreshes the OAuth token, and calls `updateQuantities` for the single item. The cron `bandcamp-inventory-push` (every 5 min) remains the safety net for any focused enqueue that fails or short-circuits. Both new tasks consult `loadFanoutGuard` (Tier 1 #1/#13 — kill switches + percentage rollout via FNV-1a deterministic correlation-id bucketing) so Phase 4 ramp posture is `fanout_rollout_percent = 0` per workspace at deploy and operators ramp 0 → 10 → 50 → 100 per integration. Workspaces without `shipstation_v2_inventory_warehouse_id` / `_location_id` set short-circuit cleanly with status `skipped_no_v2_defaults` — Phase 3 admin populates these on first successful seed. Bundle SKUs are intentionally excluded from BOTH legs (Phase 2.5(a) for v2; bundle-aggregated focused push is deferred to the cron path because the cron has the bundle math hot loop). The `external_sync_events` table is the single source of truth for "did we already push this delta to that system" — retries, replays, and webhook redeliveries collide on the unique key and become no-ops.
- **Shipping log list/detail parity invariant:** `getShipments` and `getShipmentDetail` MUST use the same fulfillment cost model (postage + materials + pick/pack), computed via `src/lib/server/shipment-fulfillment-cost.ts`. The list enriches each row with `fulfillment_total` and `fulfillment_partial`; the detail returns `costBreakdown.total` and `costBreakdown.partial`. All variant and format-cost queries MUST scope by `workspace_id`. No direct SQL to `warehouse_product_variants` or `warehouse_format_costs` for cost computation outside this module. The admin Shipping Log cost column and profitability dot show the fulfillment total (not postage-only). The dot is amber when `partial=true`; green/red when customer shipping is known and costs are complete.
- **Per-connection cutover state machine — Pass 1 substrate (Phase 3 Pass 1 — 2026-04-23):** `client_store_connections` carries `cutover_state` (`text`, NOT NULL DEFAULT `'legacy'`, CHECK ∈ {`legacy`, `shadow`, `direct`}) plus audit columns (`cutover_started_at`, `cutover_completed_at`, `shadow_mode_log_id`, `shadow_window_tolerance_seconds` bounded 30–600s). The column is **orthogonal to `do_not_fanout`** (Phase 3 §9.4 truth table); the DB CHECK `client_store_connections_cutover_dormancy_check` rejects `(cutover_state IN ('shadow','direct'), do_not_fanout=true)`. The single chokepoint `shouldFanoutToConnection()` (`src/lib/server/client-store-fanout-gate.ts`) defensively rejects unrecognized `cutover_state` values with `invalid_cutover_state` and treats `null`/`undefined` as the DB default `'legacy'`. Companion tables landed in the same migration (`20260424000002_connection_cutover_state.sql`): `connection_shadow_log` (per-push ledger that Pass 2's comparison hook will fill — `pushed_quantity` / `pushed_at` / `ss_observed_quantity` / `observed_at` / `match` / `drift_units` / `cutover_state_at_push` / `metadata`) and `connection_echo_overrides` (operator-flippable `(connection_id, override_type, is_active)` rows with a partial unique index `uq_connection_echo_overrides_active`). Pass 1 echo-override read path is `resolveShipstationV2EchoSkip(source, originatingConnectionId, supabase)` in `src/lib/server/inventory-fanout.ts`: storefront sources (`shopify`, `squarespace`, `woocommerce`, `bandcamp`) consult the override table and flip echo-skip OFF when an active row exists; direct-v2 sources (`shipstation`, `reconcile`) ALWAYS echo-skip regardless of any override (Rule #65 — pushing those back is the literal echo loop); DB lookup failures fail CLOSED (still echo-skip) so a transient error cannot silently re-introduce double-decrement. `originatingConnectionId` is plumbed through `recordInventoryChange` → `fanoutInventoryChange`, with backward-compat fallback to `metadata.connection_id` for legacy callers. Server Actions `reactivateClientStoreConnection` and `disableStoreConnection` SELECT `cutover_state` and surface actionable operator errors before hitting the DB constraint. Section C.2 of `docs/RELEASE_GATE_CRITERIA.md` documents the schema + table presence checks; `scripts/check-release-gates.sh` enforces them. Pass 1 is **GA-safe and additive** — no fanout call site changes behavior unless an operator inserts an override row (Pass 2 will gate that behind the cutover wizard at `/admin/settings/connection-cutover`).
- **Format resolution fallback chain:** When `warehouse_product_variants.format_name` is NULL, `batchBuildFormatCostMaps` falls back to `warehouse_products.product_type` via FK join (zero extra query). `product_type` is normalized through `PRODUCT_TYPE_ALIASES` (e.g. `"12\" Vinyl"` → `"LP"`) before lookup. If both are NULL or the normalized value has no cost row, the SKU is added to `unknownSkus` and `partial: true` is returned (amber dot). Migration `20260415000001_backfill_variant_format_names.sql` backfilled existing null rows: Tier 1 = SKU prefix patterns; Tier 2 = product_type alias map.
