## Core System Invariants

- UI data access uses `useAppQuery` / `useAppMutation` with `query-tiers`.
- Server-side auth context comes from `requireAuth()` and validated role checks.
- Org/workspace sensitive writes are protected by RLS or service-role actions with explicit authorization checks.
- Trigger.dev handles background and asynchronous workflows; debugging must include related tasks.
- Webhook handlers must preserve idempotency and bounded retries.
- Release confidence is enforced by `release-gate` checks and full-site audit criteria.
- Bundle inventory uses component-level tracking via `bundle_components` table. When a bundle sells (any channel), `bundle-component-fanout` decrements each component SKU. Inventory pushes compute MIN(bundle_stock, floor(component_stock/qty)) for effective availability. `bundles_enabled` flag is advisory; `bundle_components` row existence is the source of truth.
- Bandcamp follows an **authority lifecycle**: Bandcamp API is authoritative for **initial ingest** (new titles, SKU/quantity/date/price bootstrap). After staff review or physical count, the warehouse app becomes authoritative for **operational fields** (SKU, quantity, price, dates). Bandcamp remains authoritative for **descriptive/external fields** (URL, subdomain, album_title, options, sales data) permanently. Governed by `authority_status` on `bandcamp_product_mappings` (`bandcamp_initial` → `warehouse_reviewed` → `warehouse_locked`). HTML scraping (`data-tralbum`) is **enrichment only** (about, credits, tracks, package photos). Automation is **bounded** (caps, DLQ to `warehouse_review_queue`, no unbounded retry).
- **Bandcamp unmatched-create ordering + containment guardrails (dupe prevention, 2026-04-19):** `bandcamp-sync` MUST create DB rows before Shopify rows on the unmatched path: create `warehouse_products` (`shopify_product_id = NULL`) → insert `warehouse_product_variants` (must capture and handle `error`) → insert `bandcamp_product_mappings` → only then call Shopify `productSetCreate` and patch `warehouse_products.shopify_product_id`. Package-level dedup is anchored by `(workspace_id, bandcamp_item_id)` (migration `20260419000001_bandcamp_package_dedup_backstop.sql`). - **ShipStation v2 inventory transaction-type semantics (Phase 0 — Patch D2 probe, 2026-04-17):** ShipStation v2 `POST /v2/inventory` enforces a `quantity >= 1` validator on every `transaction_type` that takes a `quantity`/`new_available` field — but the validator is applied asymmetrically across operations and lifecycle stages. Empirically verified contract (full evidence: plan §5.1, §7.1.6.0, `reports/shipstation-v2-decrement-to-zero-2026-04-17.json`):
  - `increment quantity: 0` on a fresh SKU → **400** (seed cannot create a zero-stock row).
  - `increment quantity: 1+` on a fresh SKU → **200** (canonical seed path).
  - `decrement quantity: N` on tracked SKU, including landing on `available: 0` → **200**. Use this for the natural delta path; **fanout MUST use `decrement` at the 1 → 0 boundary**, no special branch.
  - `adjust quantity: 0` on tracked SKU → **200** (asymmetric vs seed; this is the documented safety net if `decrement` ever fails).
  - `modify new_available: 0` → **400** (`modify` cannot zero a SKU). **Forbidden** in any inventory write path; Phase 4 fanout unit tests must assert this combination is never emitted.
  - When tracked `available` reaches 0, ShipStation drops the row from `GET /v2/inventory?sku=…` (returns empty `inventory: []`). Reconcile (Phase 5) MUST treat "DB has `available: 0`" and "ShipStation returns no row" as equal, not as drift.

Idempotency is enforced by the `external_sync_events` ledger (`UNIQUE(system, correlation_id, sku, action)`) keyed by `(system='shipstation_v2', correlation_id='seed:{workspace_id}:{run_id}', sku, action='increment')` — Trigger.dev replays reuse the same `ctx.run.id` so duplicate inserts short-circuit and the SKU is counted as `ledger_skipped`. The task is pinned to `shipstationQueue` (concurrencyLimit 1) — same queue as `process-shipstation-shipment` so seed bursts and real-time SHIP_NOTIFY processing serialize against ShipStation v2's 60 req/min budget. `dryRun: true` evaluates the full cascade against Postgres without calling v2, surfacing gate counts to the admin Preview button. The task is bulk-shaped (Rule #59) and does NOT route through `recordInventoryChange()`; the Phase 5 reconcile sensor is the safety net for any drift introduced during seeding. Server Actions live in `src/actions/shipstation-seed.ts` and enforce `requireAuth` + `isStaff` + `workspaceId == userRecord.workspace_id`; preview polls inline up to 25s before returning a pending status (Rule #41); the real seed returns the task run id immediately and the UI polls `channel_sync_log` (`channel='shipstation_v2'`, `sync_type='seed_inventory'`) for progress (Rule #48 — never call ShipStation directly). Companion stub `shipstation-store-refresh` records refresh requests to `channel_sync_log` with `status:'deferred'` while Open Question #2 (public refresh endpoint existence) is unresolved — the day-one race fallback is the 24h auto-import + Phase 5 reconcile.

- **Rollout audit + auto-halt sensor (Phase 6 finish-line v4 — 2026-04-13):** Every change to `workspaces.fanout_rollout_percent` flows through `setFanoutRolloutPercentInternal` in `src/lib/server/admin-rollout-internal.ts`, which appends to the new `workspaces.fanout_rollout_audit jsonb NOT NULL DEFAULT '[]'` column. Rollout helper has 22 unit tests; admin UI page `/admin/settings/megaplan-ramp` is deferred (slug `megaplan-ramp-admin-page`).
- **Per-integration kill switches and percentage rollouts (Tier 1 hardening #1, #13 — 2026-04-13):** `workspaces` carries five guard columns: the existing global `inventory_sync_paused`, plus per-integration `shipstation_sync_paused`, `bandcamp_sync_paused`, `clandestine_shopify_sync_paused`, `client_store_sync_paused` (each with companion `*_paused_at` / `*_paused_by` audit columns), and `fanout_rollout_percent smallint NOT NULL DEFAULT 100 CHECK (BETWEEN 0 AND 100)`. The single chokepoint is `loadFanoutGuard(supabase, workspaceId)` + `guard.shouldFanout(integration, correlationId)` in `src/lib/server/fanout-guard.ts`. `inventory-fanout.ts` (the central inventory fanout entry point) MUST consult the guard before each integration push (Clandestine Shopify, client stores, Bandcamp, ShipStation v2 fanout when wired in Phase 4). Rollout bucketing is deterministic: `correlationIdBucket(correlationId)` uses 32-bit FNV-1a so a given correlation_id always lands in the same 0..99 bucket — replays with the same `webhook_id` / `task_run_id` re-evaluate identically and never half-emit the same event. `fanout_rollout_percent = 100` is a no-op (everything passes); `0` halts everything; values in between gate by `bucket < percent`. Global pause is checked first; integration-specific pause next; rollout last. Operator workflow: flip a single column to pause one integration without halting the rest of the system, or set `fanout_rollout_percent = 10` to canary a new fanout pipeline against 10% of traffic before flipping back to 100. Guard caches per-call only — no in-process memoization, so a workspaces-row update takes effect on the next fanout invocation.
- **External sync events retention is automated (Tier 1 hardening #14 — 2026-04-13):** `external-sync-events-retention` (cron `30 3 * * *` America/New_York) prunes `external_sync_events` to bound table growth: rows with `status='success'` and `completed_at < now() - interval '7 days'` are deleted; rows with `status='error'` are kept 30 days for forensics. The cleanup is backed by partial index `idx_external_sync_events_completed_status (status, completed_at) WHERE completed_at IS NOT NULL` (migration `20260413000010_tier1_hardening.sql`), so the daily delete is index-only and never table-scans. Idempotent — re-running the same day is a no-op against the empty result set. The ledger remains the source of truth for in-flight idempotency (Phase 0.5 invariant); only fully terminal rows are pruned.
- **Daily reconciliation report (Tier 1 hardening #11 — 2026-04-13):** `daily-recon-summary` (cron `0 12 * * *` UTC) aggregates the prior 24h of operational health into a single structured log line and (if `OPS_ALERT_EMAIL` is set) a plaintext email digest via Resend: `external_sync_events` success/error counts grouped by system, `warehouse_review_queue` open items grouped by severity, and `channel_sync_log` failure/partial counts grouped by channel. - **Weekly backup verification probe (Tier 1 hardening #8 agent slice — 2026-04-13):** `weekly-backup-verify` (cron `0 9 * * 0` Sundays UTC) is a sensor-grade liveness check on the critical-table set (`workspaces`, `warehouse_inventory_levels`, `warehouse_product_variants`, `warehouse_orders`, `external_sync_events`). For each table it asserts non-zero `count(*)` and verifies the most-recent timestamp column (`created_at` / `updated_at` / `started_at`) is within a per-table freshness threshold (orders ≤ 14 days, sync events ≤ 7 days, etc.). Stale or empty tables emit alert-level log lines that the SLO doc maps to PagerDuty. The agent task does NOT exercise full sandbox restore — that is operator-led and documented in `docs/operations/runbooks/backup-verify.md`. Together, the agent probe (aliveness) and the operator runbook (restorability) constitute the full Tier 1 #8 contract.
- **Service-role usage is documented and CI-enforced (Tier 1 hardening #2 — 2026-04-13):** every `createServiceRoleClient()` call site in `src/**` is enumerated in `docs/security/service-role-usage.md` with a justification (Trigger task / webhook / OAuth handler / Server Action / shared lib). `scripts/check-service-role-usage.sh` (CI) greps the source tree for `createServiceRoleClient` (excluding tests + the helper itself), counts unique files, and fails the build if the count drifts from the manifest entry count. - **Phase 4 bidirectional bridge (2026-04-13):** Per-SKU cross-system inventory mirroring runs through two ledger-gated, queue-pinned tasks instead of inline calls. (1) **`shipstation-v2-decrement`** (`shipstationQueue`, concurrencyLimit 1): triggered by `bandcamp-sale-poll` after each successful `recordInventoryChange()` for a Bandcamp sale. Correlation-id shape `sale:{band_id}:{package_id}:{newSold}:{sku}`. Skip cascade: `fanout-guard.shouldFanout("shipstation", correlationId)` → `workspaces.shipstation_v2_inventory_warehouse_id`/`_location_id` IS NOT NULL → variant resolved → variant is NOT a `bundle_components.bundle_variant_id` (Phase 2.5(a) bundle exclusion) → ledger acquired (`UNIQUE(system='shipstation_v2', correlation_id, sku, action='decrement')`). On the success path, calls `adjustInventoryV2({ transaction_type:'decrement', quantity:|delta| })` — NEVER `modify` (Phase 0 Patch D2 — `modify new_available:0` returns 400). On failure, marks the ledger row `error` and re-throws so Trigger.dev retries the task (the same correlation_id makes the retry a no-op once the prior attempt eventually succeeds elsewhere). (2) **`bandcamp-push-on-sku`** (`bandcampQueue`, concurrencyLimit 1, OAuth-serialized — Rule #9): triggered by `process-shipstation-shipment` after each per-line successful `recordInventoryChange()`. Correlation-id shape `ship:{shipmentId}:{sku}`. Skip cascade: `fanout-guard.shouldFanout("bandcamp", correlationId)` → variant resolved → NOT distro (`warehouse_products.org_id IS NOT NULL`) → variant is NOT a bundle parent → `bandcamp_product_mappings` row exists → `push_mode IN ('normal','manual_override')` (Phase 1 contract) → mapping is package-level (option-level mappings defer to the 5-min `bandcamp-inventory-push` cron) → ledger acquired (`UNIQUE(system='bandcamp', correlation_id, sku, action='modify')`). On the success path, computes `pushed_quantity = MAX(0, available - effective_safety)` from `warehouse_inventory_levels` (identical formula to the cron path so cron + focused push agree), refreshes the OAuth token, and calls `updateQuantities` for the single item. The cron `bandcamp-inventory-push` (every 5 min) remains the safety net for any focused enqueue that fails or short-circuits. Both new tasks consult `loadFanoutGuard` (Tier 1 #1/#13 — kill switches + percentage rollout via FNV-1a deterministic correlation-id bucketing) so Phase 4 ramp posture is `fanout_rollout_percent = 0` per workspace at deploy and operators ramp 0 → 10 → 50 → 100 per integration. Workspaces without `shipstation_v2_inventory_warehouse_id` / `_location_id` set short-circuit cleanly with status `skipped_no_v2_defaults` — Phase 3 admin populates these on first successful seed. Bundle SKUs are intentionally excluded from BOTH legs (Phase 2.5(a) for v2; bundle-aggregated focused push is deferred to the cron path because the cron has the bundle math hot loop). The `external_sync_events` table is the single source of truth for "did we already push this delta to that system" — retries, replays, and webhook redeliveries collide on the unique key and become no-ops.
- **Shipping log list/detail parity invariant:** `getShipments` and `getShipmentDetail` MUST use the same fulfillment cost model (postage + materials + pick/pack), computed via `src/lib/server/shipment-fulfillment-cost.ts`. The list enriches each row with `fulfillment_total` and `fulfillment_partial`; the detail returns `costBreakdown.total` and `costBreakdown.partial`. All variant and format-cost queries MUST scope by `workspace_id`. No direct SQL to `warehouse_product_variants` or `warehouse_format_costs` for cost computation outside this module. The admin Shipping Log cost column and profitability dot show the fulfillment total (not postage-only). The dot is amber when `partial=true`; green/red when customer shipping is known and costs are complete.
- **Format resolution fallback chain:** When `warehouse_product_variants.format_name` is NULL, `batchBuildFormatCostMaps` falls back to `warehouse_products.product_type` via FK join (zero extra query). `product_type` is normalized through `PRODUCT_TYPE_ALIASES` (e.g. `"12\" Vinyl"` → `"LP"`) before lookup. If both are NULL or the normalized value has no cost row, the SKU is added to `unknownSkus` and `partial: true` is returned (amber dot). Migration `20260415000001_backfill_variant_format_names.sql` backfilled existing null rows: Tier 1 = SKU prefix patterns; Tier 2 = product_type alias map. Post-migration result: 418 genuinely ambiguous variants remain NULL (correctly → amber). `T-Shirt` and `Other` format rows are missing from `warehouse_format_costs` and need to be added in admin for those 4 variants to show green/red.
