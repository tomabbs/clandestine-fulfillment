## Method

Binary-step concurrency through `[2, 5, 10, 15, 20, 30, 40, 50]`. Per step: 10s burst at that concurrency, then 60s recovery before the next step. Auto-cleanup of `webhook_events` rows after each step (zero leakage to recovery sweep).

Identical hot path as Sub-pass A.1: real Shopify-shaped, HMAC-signed `inventory_levels/update` payloads against the production Node ingress route, targeting Northern Spy in legacy state so the downstream task is a no-op.

## Two material discoveries beyond "the ceiling exists"

### Discovery 1 — Sustainable ceiling ≈ 27.5 enqueues/sec @ concurrency 10

This is the operational number to plan against. Anything below this rate is silent and reliable; anything above it triggers HRD-17.1 graceful degradation (which works, but adds 5-min latency to fanout via the recovery sweeper).

**Real-world headroom analysis:**

| Scenario | Estimated peak rps | vs ceiling |
|---|---:|:---:|
| Single Shopify shop, normal sale velocity | ~0.1 rps | **safe** |
| Single Shopify shop, label-day pre-order drop | ~5 rps | **safe** |
| Single shop, bulk SKU edit (50 SKUs) | ~10 rps over 5s = 2 rps | **safe** |
| Three shops simultaneously editing 20 SKUs each | ~12 rps over 5s | **safe** |
| Five shops × bulk inventory rebuild (200 SKUs each) | ~50 rps over 20s | **above ceiling** |
| 10 shops × Shopify Flow automation in same minute | ~30+ rps sustained | **at ceiling** |

The **multi-shop simultaneous bulk-edit scenario** is the realistic failure case, not single-shop bursts.

### Discovery 2 — Penalty-box behavior at concurrency ≥20

At concurrency 20 and above, `ms_to_first_503 = 0` — meaning the very first request of the step receives a 503 before any other step-N request even lands. This persisted for steps 5, 6, 7, AND 8, despite a 60s recovery wait between each.

This is not consistent with a simple "leaky bucket" rate-limit (where 60s of refill should restore some capacity). It's consistent with one of:

- **Long rate-limit window** (e.g. 5-minute rolling window) where a single bad step consumed quota for the next several steps.
- **Punishment cooldown** where exceeding the limit triggers a longer block than just "wait for the bucket to refill."
- **Account-level circuit breaker** where Trigger.dev backs off the org for an extended window once heavily abused.

**Critical operational implication:** a single bad burst doesn't just delay that burst — it locks out subsequent legitimate webhooks for an extended window. The HRD-17.1 recovery sweep handles the legitimate retry path, but during that window real-time fanout is broken.

This finding **changes the threat model** for Phase 4. We're not just protecting against "rare burst exceeds ceiling and we accept the 5-min recovery sweep delay" — we're protecting against "rare burst exceeds ceiling AND triggers a multi-minute cascade where every subsequent webhook (legitimate or not) falls into the same recovery sweep window."

## Implications for the Phase 4 architecture decision

The original X-1 framing ("does Edge migration improve cold-start ingress p95?") is still valid but **insufficient on its own**. Two facts now coexist:

1. **Edge migration alone is not enough.** Even with sub-100ms Edge cold-start, every request still synchronously calls `tasks.trigger()`. The ceiling moves from "Node-route-bound (which we proved)" to "Trigger-enqueue-bound (which we just measured)" — same 27.5 rps, same penalty-box behavior. Edge gets us better cold-start latency but does nothing about the enqueue ceiling.

2. **`waitUntil()` decoupling (already in §9.5 deliverable 2) is now mandatory, not optional.** Returning 200 OK before the Trigger POST means Shopify never sees the 503 even if the enqueue fails — Shopify thinks the webhook landed cleanly, and the recovery sweep handles the actual enqueue out-of-band. **But** this still leaves real-time fanout broken during the penalty-box window.

3. **Real fix requires removing `tasks.trigger()` from the per-webhook critical path.** Three viable architectures:

   | Option | What it changes | Effort | Removes Trigger from hot path? |
   |---|---|---|---|
   | **A. `tasks.batchTrigger()`** | Aggregate webhooks landing within 100ms window into one batch call | Small (1-2 days) | No, but reduces call rate ~10× |
   | **B. `waitUntil()` only** | Decouple 200 OK from Trigger POST | Small (already in §9.5 deliverable 2) | No — still calls Trigger per webhook, just async |
   | **C. Postgres NOTIFY → consumer** | Long-running Node consumer LISTENs for `webhook_events` inserts; never calls Trigger from the route | Medium (1-2 weeks; needs Fly/Railway/Render hosting for the consumer) | **Yes** — Trigger is no longer on the per-webhook path |

   **Combining A + B** gets us most of the way at low cost. We process 27.5 enqueues/sec ÷ 10× batch = 275 logical webhooks/sec ceiling, and even when the batch enqueue fails, Shopify already got its 200 OK.

   **Option C** is the only architecture that removes the dependency entirely, but it adds an always-on hosting surface (the consumer process) and a NOTIFY/LISTEN durability story (what if the consumer is down for 10 min?). For our scale, A+B is probably sufficient unless Shopify changes its delivery model.

## Open questions deferred to Phase 4 build

1. **What is Trigger.dev's actual rate-limit window?** Re-run probe with `--ladder=20,30 --recovery-seconds=300` (and again with `--recovery-seconds=600`) to localize the penalty-box duration. Required for sizing the `waitUntil()` retry budget.
2. **Does `tasks.batchTrigger()` count as 1 request or N requests against the rate limit?** Answers whether Option A above is a 10× win or a wash. Test via a small probe: `tasks.batchTrigger([10 items])` vs `10 × tasks.trigger(item)` from a clean baseline.
3. **What's the per-shop-per-second realistic burst from Shopify?** Pull historical `webhook_events` data for the 95th-percentile per-shop burst window. If it's <10/sec, the ceiling concerns are entirely about multi-shop simultaneity, which we can handle with per-shop queuing at the route level.

These are recorded under the `phase4-trigger-enqueue-saturation` deferred follow-up for Phase 4 build planning.

## Bottom line

- **Ceiling**: 27.5 enqueues/sec at concurrency 10, with hard cliff at concurrency 15.
- **Recovery**: ≥60s; suspected 5-min penalty-box window for severe bursts.
- **F-7 / X-1 verdict** unchanged — Run #1's data already proves Edge migration is needed for cold-start latency.
- **X-1.b verdict** is new and decisive — Edge migration alone won't fix the enqueue ceiling. Phase 4 build scope must include `waitUntil()` decoupling (already in §9.5 deliverable 2) AND should evaluate `tasks.batchTrigger()` (cheap, ~10× headroom) before committing to the heavier Postgres NOTIFY architecture.
