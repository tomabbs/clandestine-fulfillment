This is Run #1 of that two-run gate. Method:

- Target: live production Node ingress at

- Mode: 50 concurrent connections, sustained for 60s (matches plan

- Warmup: 30s sleep before run start to give Vercel lambdas time to

## Critical finding — bottleneck is NOT Node ingress

The 503 body is `{"ok":false,"status":"enqueue_failed","will_retry":true}`.
This is **HRD-17.1 graceful degradation working as designed** — not
Vercel rate-limiting or Node cold-start failure. The route handler
sequence is:

1. `req.text()` + HMAC verify ✅ (median ~30 ms)
2. Freshness + JSON parse ✅
3. `INSERT INTO webhook_events (status='received')` ✅
4. **`tasks.trigger("process-client-store-webhook", { idempotencyKey })`** ← the call that's failing
5. On failure: `UPDATE status='enqueue_failed'`, return 503

The fast 503 latency (median 225 ms) confirms the Node route is healthy
— it's the **Trigger.dev v4 enqueue API** that's saturating. Once that
saturates, every subsequent request tags its `webhook_events` row as
`enqueue_failed` and the recovery sweeper
(`webhook-events-recovery-sweep`, every 5 min) re-fires the enqueue
later. **No webhook is dropped on the floor.** Shopify also retries on
503 with backoff.

Timing: first 503 fired at startedAt=4,475 ms (i.e. ~4.5s into the
60s burst). The first ~5s of the run was healthy — Trigger's enqueue
API absorbed ~646 requests cleanly — then it hit the rate limit and
the rest of the run got the fast-fail path.

## What this means for F-7

Two distinct failure modes surfaced; X-1 only addresses one:

### Failure mode 1 — cold-start ingress latency (F-7 territory)

- **200-only p95 = 1,507 ms** vs F-7 budget of 800 ms. ~2x over.
- Cold-start proxy p95 = 1,660 ms (slightly worse because the
  cold-start sample is the first-wave of 50 simultaneous requests,
  which are all competing for the same 0→N lambda boot).
- Edge migration (Vercel Edge / Cloudflare Workers) WOULD fix this.
  Edge cold-start is ~50–100 ms vs ~1,500 ms for Node.

### Failure mode 2 — Trigger.dev enqueue throughput (NEW — not in X-1)

- 77% enqueue-failed at 50 concurrent.
- This is **independent** of the ingress runtime. Even with 50ms
  Edge ingress, each request still has to call `tasks.trigger()`, and
  `tasks.trigger()` is the throttled component.
- HRD-17.1 protects us — the recovery sweeper guarantees no data loss
  — but during the burst window, real-time fanout to client stores
  is delayed by up to 5 minutes.
- Mitigations to consider (NOT for this pass — surfaced for future
  planning):
  - Bulk-enqueue: aggregate webhooks landing within a 100ms window
    into a single Trigger run that processes N events.
  - Direct Postgres NOTIFY → consumer process (skip Trigger entirely
    for high-volume events).
  - Trigger.dev v4 batch API (`tasks.batchTrigger()`) — already
    available.

## What this means for X-1 decision

- F-7 cold-start fail is **definitive** — 1,507 ms p95 is not within
  measurement noise of 800 ms. Run #2 in 24h is unlikely to flip this.
- The 503 saturation finding is **decision-shifting** — it suggests
  Phase 4's scope should expand beyond "make ingress faster" to
  "remove the Trigger enqueue from the critical path."
- Recommendation: still take Run #2 in 24h+ for X-1 audit-trail
  completeness, but Phase 4 planning should incorporate the
  bulk-enqueue / NOTIFY findings now rather than wait.
